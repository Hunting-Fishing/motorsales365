
# 365motorsales Launch-Readiness Plan (in-house, zero external secrets)

Constraint: no new external API keys / shared secrets requested from the user. Anything that previously required `PAYMONGO_WEBHOOK_SECRET`, `XENDIT_CALLBACK_TOKEN`, `OPS_DIGEST_CRON_KEY` is replaced with in-app primitives. The Stripe gateway secrets (`STRIPE_SANDBOX_API_KEY`, `STRIPE_LIVE_API_KEY`, `PAYMENTS_*_WEBHOOK_SECRET`, `LOVABLE_API_KEY`, `SUPABASE_SERVICE_ROLE_KEY`) are already auto-provisioned by Lovable Cloud — no action.

Work runs in 4 phases. Each is independently shippable.

---

## Phase 1 — P0 Security & Correctness

### 1.1 Lock down `/api/public/payment-events`
- **Drop external-provider support entirely.** The only real money rail is Stripe, and Stripe lifecycle already flows through `/api/public/payments/webhook` with verified `stripe-signature`. The `payment-events` route only exists for admin tooling fan-out, so:
  - Remove the `PAYMONGO_WEBHOOK_SECRET` / `XENDIT_CALLBACK_TOKEN` path.
  - Replace `x-debug-token` with an **internal HMAC**: server signs the request body with a key stored in a new private DB table `internal_webhook_keys(name TEXT PK, secret TEXT, rotated_at TIMESTAMPTZ)`. Secret is generated by a migration via `gen_random_bytes(32)` → hex; never returned to the client; only readable via `service_role`.
  - Verifier reads `internal_webhook_keys.secret WHERE name='payment_events'` using `supabaseAdmin` and compares `x-internal-signature` (HMAC-SHA256 over raw body) with `timingSafeEqual`.
  - Add a typed enqueuer `enqueuePaymentEvent()` in `src/lib/payment-events.server.ts` that admin code calls directly (in-process) — eliminates the HTTP hop for internal callers. The HTTP route stays for future cron use.

### 1.2 Server-side admin guard
- Create `src/integrations/supabase/admin-middleware.ts` exporting `requireAdminRole` — chains `requireSupabaseAuth`, then checks `has_role(userId, 'admin')` via `supabaseAdmin`. Throws `Response('Forbidden', { status: 403 })` otherwise.
- Apply to every destructive admin server fn: `admin-users.functions.ts`, `admin.reports.tsx` mutations, `business-pages.functions.ts` admin paths, `education.functions.ts` admin paths, `ads.functions.ts`, referrals/pricing/currencies/redemptions/type-suggestions/verifications admin mutations.
- Client `effectiveRoles` stays — UI-hide only.

### 1.3 Stripe gateway resilience
- Wrap `createStripeClient`'s `fetch` with retry: 2 attempts on 5xx / network error, exponential backoff (250 ms, 1 s), skip retry for non-idempotent POSTs without `Idempotency-Key`.
- After final failure call `alertOps('stripe.gateway_unreachable', …, { severity: 'critical' })`.
- Document single-track gateway in `docs/STRIPE_GOLIVE.md`.

### 1.4 Service worker version stamping
- Replace hardcoded `VERSION = "v1"` in `public/sw.js` with `__SW_VERSION__` injected by a tiny Vite define plugin reading `package.json.version + Date.now()`-shorthash.
- `activate` deletes any cache key not matching current version.

### 1.5 Type-safety sweep on money paths
- Remove `as any` casts in: `src/utils/payments.functions.ts`, `src/routes/api/public/payments/webhook.ts`, `src/lib/listing-payment.functions.ts`, `src/lib/boosts.functions.ts`, `src/lib/business-subscriptions.functions.ts`.
- Use generated Supabase types + `Stripe.*` types; narrow webhook payloads with Zod schemas.

**Phase 1 verification**: Stripe sandbox webhook returns 200; `/api/public/payment-events` returns 401 without `x-internal-signature` and 200 with; calling an admin server fn as a non-admin returns 403; SW activate logs new version after publish.

## Technical Notes — in-house security primitives

- **No new secrets requested.** Everything that previously needed a `process.env.*` secret is replaced by a row in a service-role-only table:
  - `public.internal_webhook_keys(name TEXT PK, secret TEXT, rotated_at TIMESTAMPTZ DEFAULT now())`
  - `public.internal_cron_tokens(job_name TEXT PK, token TEXT, rotated_at TIMESTAMPTZ DEFAULT now())`
  - Both tables: `GRANT ALL ... TO service_role` only; RLS enabled with **no** policies (default-deny for `anon` / `authenticated`).
  - Migration seeds initial values via `encode(gen_random_bytes(32),'hex')`. Admin UI (`/admin/sandbox`) gets a "Rotate" button that calls a `rotate_internal_secret` server fn (admin-guarded) which UPSERTs a fresh value.
- **Site-wide config** moves from hardcoded env to `public.site_settings.app_url` (already exists — verify and use).
- `requireAdminRole` and the new HMAC/token verifiers live in `src/integrations/supabase/` so they share the existing `supabaseAdmin` import.
- Stripe gateway retries + alertOps escalation live inside `src/lib/stripe.server.ts` — no new file.
- SW version stamping uses Vite `define` only — no new build secrets.
